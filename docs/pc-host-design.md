# PC Host 上位机设计说明

## 1. 目标

为 S800 / TM4C1294 大作业提供一个基于 PyQt5 的 PC 上位机，代码放在 `pc_host/` 目录。

设计目标：
- 以老师验收演示为主，同时保留基础调试能力
- 主界面直观展示“控制 + 数字孪生镜像 + 日志”三部分
- 覆盖课程要求中的核心协议与按键映射
- 支持核心功能 P1-P5 和扩展功能 E1-E4
- 保持作业风格，不过度工程化，不过度堆砌异常恢复机制

## 2. 运行环境与依赖

- Python 版本：3.11.9

## 3. 协议基线

上位机以以下命令总表为准。

### 3.1 核心命令

- `*RST`
  - 复位时钟 / 日期 / 闹钟，`DISPLAY=ON`，`FORMAT=LEFT`
- `*SET:DATE YEAR / MONTH / DATE`
  - 单参或任意组合
- `*SET:TIME HOUR / MINUTE / SECOND`
  - 单参或任意组合
- `*SET:ALARM HOUR / MINUTE / SECOND / OFF`
  - 单参或任意组合
- `*SET:DISPLAY ON / OFF`
  - 数码管亮灭
- `*SET:FORMAT LEFT / RIGHT`
  - 显示与应答方向
- `*SET:MSG <text>`
  - 长度不超过 32 字节，保留原大小写
- `*SET:BEEP <MS>`
  - 10-5000 ms 蜂鸣器响
- `*SET:LED <hex2>`
  - 直接设置 LED 字节
- `*SET:KEY <NAME>`
  - 模拟按键，`<NAME>` 只能取 10 个协议键名
  - 模拟执行后不再上报 `*EVT:KEY`，避免 PC 与板端形成环回
- `*SET:MODE DAY / NIGHT`
  - 昼夜模式
- `*GET:DATE / *GET:TIME / *GET:ALARM / *GET:DISPLAY / *GET:FORMAT`
  - 参数与 `SET` 一致或省略
- `*PING`
  - 心跳，板端应答 `*PONG <uptime_s>`

### 3.2 协议键名

`*SET:KEY` 与 `*EVT:KEY` 的 `<NAME>` 取值固定为 10 项：
- `FUNC`
- `SHIFT`
- `ADD`
- `SAVE`
- `DISP`
- `SPEED`
- `FORMAT`
- `EXT`
- `USER1`
- `USER2`

除这 10 项外，其余键名均视为非法协议值。

## 4. 串口通信策略

上位机不能把串口当作“点一下立刻无节制发送”的通道，而要统一调度。

### 4.1 命令发送节流

所有串口命令都进入统一发送队列，执行以下规则：
- 相邻两条命令最小间隔 **500ms**
- 这个节流对以下所有场景都生效：
  - 初始化同步 `GET`
  - 用户点击按钮触发的 `SET`
  - 周期性 `PING`
  - `SET` 成功后的补 `GET`
  - 扩展功能自动下发的命令

目标是降低单片机在高频收包时出现异常的概率。

### 4.2 ready 门控

上位机连接串口后，不能立刻发送 `*GET` 命令，必须等单片机 ready。

当前设计约束如下：
- 先进入 `waiting_ready` 状态
- ready 之前，不发送初始化 `GET`
- 如果固件后续提供显式 READY 事件或 READY 应答，则以上位机收到的 READY 为准
- 如果当前固件没有显式 READY 信号，则首版采用 **boot guard + PING** 的方式：
  - 默认等待约 10 秒启动保护时间
  - 再发送 `*PING`
  - 收到 `*PONG` 后，开始初始化同步

当前固件启动动画相关时间常量总计约 9.5 秒，因此 10 秒保护时间能覆盖首版需求。

### 4.3 初始化同步顺序

连接成功并 ready 后，上位机按以下顺序同步初始状态，每条命令间隔 500ms：

1. `*PING`
2. `*GET:DISPLAY`
3. `*GET:FORMAT`
4. `*GET:DATE`
5. `*GET:TIME`
6. `*GET:ALARM`

说明：
- `MODE` 当前不在初始化 `GET` 中，因为命令总表没有 `*GET:MODE`
- `LED` 当前不在初始化 `GET` 中，因为命令总表没有 `*GET:LED`
- 初始化阶段以命令总表为准，不依赖事件历史

## 5. 总体界面方案

采用单窗口多分区的 **A2 布局**：

- **顶部状态栏**：显示串口连接状态、当前串口、ready 状态、心跳/延迟、FORMAT、MODE、ALARM 状态
- **左侧控制区**：负责串口管理、基础时钟控制、显示控制、外设控制、虚拟按键、扩展功能入口、协议演示入口
- **右侧主显示区**：放数字孪生镜像面板，作为主视觉区域
- **底部整行区域**：放收发日志和简化原始命令发送区

选择 A2 的原因：
- 右侧镜像区更大，适合演示“软件镜像跟随硬件变化”
- 左侧控制操作集中，老师更容易理解交互流程
- 底部日志区能证明串口收发真实发生，不是纯静态演示

## 7. 状态同步策略

整体数据流采用“统一发送队列、初始化 GET 建状态、运行期 EVT 增量更新、本地 shadow 补足缺口”的方式。

### 7.1 初始同步

连接后：
- 先等 ready
- 再按固定顺序执行初始化 `GET`
- 以返回值建立首版状态

这一步不能依赖 `*EVT:*`，因为上位机打开时没有事件历史。

### 7.2 运行期同步

运行期的状态更新来源分三类：

1. **GET 可查询状态**
   - `DATE`
   - `TIME`
   - `ALARM`
   - `DISPLAY`
   - `FORMAT`

2. **EVT 增量状态**
   - 若固件提供 `*EVT:*`，优先用于增量更新界面

3. **本地 shadow 状态**
   - 对当前无 `GET` 能力的状态，用“最近一次成功下发值”作为已知状态
   - 例如 `MODE`、部分 LED 状态

### 7.3 写后补读策略

对于可查询的命令族，采用“`SET` 成功后补 `GET`”的策略：
- `SET:DATE` 后补 `GET:DATE`
- `SET:TIME` 后补 `GET:TIME`
- `SET:ALARM` 后补 `GET:ALARM`
- `SET:DISPLAY` 后补 `GET:DISPLAY`
- `SET:FORMAT` 后补 `GET:FORMAT`

这样可以避免 UI 只靠本地猜测状态。

### 7.4 当前状态可信度分层

首版状态按可信度分为三层：

- **可真实同步**
  - `DISPLAY`
  - `FORMAT`
  - `DATE`
  - `TIME`
  - `ALARM`

- **可用 shadow 维护**
  - `MODE`
  - 由 PC 自己刚设置过的 `LED`

- **当前无法可靠开机同步**
  - 7SEG 当前内容
  - 小数点状态
  - 板端当前 LED 真值
  - 消息滚动页状态
  - 需要新增事件后才能完全镜像的显示态细节

## 9. 按键映射分析

上位机与固件都要以以下按键表为准。

### 9.1 按键完整映射

- `FUNC`：K1
  - 短按：编辑模式切换；响铃中关闹钟
  - 长按：保存并退出（同 `SAVE`）
- `SHIFT`：K2
  - 短按：编辑中切换高亮字段
- `ADD`：K3
  - 短按：当前字段 +1
  - 长按：连加，频率不低于 5Hz
- `SAVE`：K4
  - 短按：保存并退出编辑
- `DISP`：K5
  - 短按：主显示切换，时间 / 日期 / 年份
- `SPEED`：K6
  - 短按：流水速度 2 级切换
- `FORMAT`：K7
  - 短按：流水方向切换，等价于 `*SET:FORMAT`
- `EXT`：K8
  - 短按：预留键，仅触发 `*EVT:KEY EXT`
- `USER1`：GPIO
  - 短按：请求 PC 对时，触发 `*EVT:KEY USER1`
- `USER2`：GPIO
  - 短按：数码管短显天气 5 秒，前提是 PC 已下发天气数据

### 9.2 上位机处理原则

- 虚拟按键区只发这 10 个键名
- 虚拟按键默认只模拟短按功能
- 长按功能不默认映射为普通点击；若要演示长按，需要单独做“长按按钮”或“重复发送策略”
- `USER1` 事件由 PC 端触发 NTP 对时流程
- `USER2` 事件由板端触发天气短显；PC 负责预先提供天气数据并监听显示变化
- `EXT` 不应带有页面切换或状态副作用

## 11. 扩展功能设计

### 11.1 E1 网络对时

实现思路：
- 点击 NTP 按钮时，通过 `ntplib` 获取标准时间
- 将结果转换为日期 / 时间命令下发给 S800
- 所有命令通过统一发送队列下发
- 界面显示最近一次对时成功或失败状态
- 收到 `*EVT:KEY USER1` 时，自动触发与按钮相同的流程

### 11.2 E2 天气获取

实现思路：
- 用 `requests` 获取本地天气
- 解析出温度与天气状况
- 按协议转换后下发给 S800
- PC 端界面显示最近一次天气结果
- 收到 `*EVT:KEY USER2` 后，观察板端天气短显的镜像变化

### 11.3 E3 自动昼夜模式

实现思路：
- 使用 `astral` 计算本地日出日落时间
- 周期性判断当前应处于 DAY 还是 NIGHT
- 自动发送 `*SET:MODE DAY` 或 `*SET:MODE NIGHT`
- 若当前固件仍无 `GET:MODE`，则界面中的 MODE 以最近一次成功下发值为准

### 11.4 E4 数据可视化看板

实现思路：
- 将闹钟、对时、按键、模式切换等事件保存到本地文件
- 用 `matplotlib` 画简单统计图
- 以“轻量看板”方式展示，不做完整数据库系统

## 12. 固件对齐情况与待修正项

本节用于明确：上位机设计以命令总表和按键映射表为准；如果当前固件与其不一致，则需要补充或修正固件。

### 12.1 当前已对齐项

- `*RST` 存在
- `*PING -> *PONG <uptime_s>` 存在
- `SET/GET:DATE` 基本存在
- `SET/GET:TIME` 基本存在
- `SET/GET:ALARM` 基本存在
- `SET/GET:DISPLAY` 基本存在，当前固件代码中用的是 `DISPlay` 匹配形式
- `SET/GET:FORMAT` 基本存在
- `SET:BEEP` 存在，但语义是蜂鸣时长，不是频率
- `SET:LED` 存在
- `SET:MODE` 存在
- `SET:KEY` 当前存在，但合法键名约束尚未收紧
- `*EVT:KEY` 与 `*EVT:KEYRAW` 当前已见

### 12.2 当前缺口与修正建议

#### A. 缺少显式 READY 机制

当前固件没有明确的 UART READY 信号。

建议：
- 固件后续可补一个 READY 事件或 READY 应答
- 若不补，则 PC 首版继续使用 10 秒 boot guard + `*PING` 探活

#### B. 缺少 `GET:MODE`

当前命令总表没有 `GET:MODE`，这会导致上位机无法在开机后真实同步 MODE。

建议：
- 若希望 MODE 开机可同步，补 `*GET:MODE`
- 或补 `*EVT:MODE`

#### C. 数字孪生依赖的显示类事件不完整

若要完整实现 7SEG / LED / MODE 镜像，固件还需具备：
- `*EVT:DISP`
- `*EVT:LED`
- `*EVT:MODE`

若这些事件缺失，则首版镜像只能做到部分可信同步。

#### D. `*SET:KEY` 需要限制合法键名

当前固件应只接受以下 10 个协议键名：
- `FUNC`
- `SHIFT`
- `ADD`
- `SAVE`
- `DISP`
- `SPEED`
- `FORMAT`
- `EXT`
- `USER1`
- `USER2`

不应接受任意字符串。

#### E. `USER1 / USER2` 需要改为独立语义

当前固件中，`USER1 / USER2` 若被实现为 `TIME/DATE` 或 `FUNC/SAVE` 的别名，则与协议表不一致。

建议：
- `USER1` 只表示“请求 PC 对时”
- `USER2` 只表示“请求天气短显”
- 不再与其他键名混用

#### F. `EXT` 需要改为无副作用预留键

`EXT` 只应触发 `*EVT:KEY EXT`，不应承担页面切换或消息页切换逻辑。

#### G. `SPEED` 需要按 2 级切换实现

协议要求：
- `SPEED` 为 2 级切换

如果当前固件是 3 档循环，则需要修正为 2 档。

#### H. 编辑态与长按行为需要补齐

协议映射要求的以下语义若当前固件未实现，则需补齐：
- `FUNC` 短按：编辑模式切换；响铃中关闹钟
- `FUNC` 长按：保存并退出
- `SHIFT`：切换高亮字段
- `ADD` 短按：当前字段 +1
- `ADD` 长按：不低于 5Hz 连加
- `SAVE`：保存并退出编辑
- `DISP`：切换时间 / 日期 / 年份显示

#### I. 需要核对 K1-K8 的实体键顺序映射

按键完整映射表给出的顺序是：
- `FUNC` `SHIFT` `ADD` `SAVE` `DISP` `SPEED` `FORMAT` `EXT`

若当前固件扫描顺序与该表不一致，则需核对接线与事件映射关系，保证 `*EVT:KEY <NAME>` 与物理键位一致。

## 13. 异常处理策略

按照课程作业风格，采用“可见、不中断、不过度设计”的处理方式。

- **串口占用 / 无法打开**
  - 弹窗提示
  - 日志区红色高亮
  - 保持 GUI 可用

- **串口断开 / 读取失败**
  - 状态栏改为断开
  - 停止继续调度后续串口命令
  - 日志记录断开原因

- **ready 前发送命令**
  - 阻止发送
  - 在日志区提示“设备未 ready”

- **协议解析异常**
  - 保留原始报文
  - 日志中标记解析失败
  - 不让主窗口崩溃

- **板端 `ERROR ...` 应答**
  - 日志区高亮
  - 若由用户按钮触发，则可附加短弹窗提示

- **网络超时 / 天气或 NTP 失败**
  - 在扩展区或日志区提示失败
  - 不影响本地串口控制继续使用

不做内容：
- 多级自动重试
- 复杂断线重连策略
- 后台任务编排系统
- 统一异常上报平台

## 15. 实现边界

本设计刻意保持课程作业风格，边界如下：

- 不做复杂插件系统
- 不做数据库层
- 不做自动更新
- 不做跨平台打包方案设计
- 不做高可靠后台守护
- 不做与板端脱离的独立仿真器

重点是把“上位机控制 + 数字孪生镜像 + 扩展演示”这条主线做好。

## 16. 实施顺序建议

建议开发顺序：

1. 搭主窗口骨架和基础布局
2. 完成串口管理、ready 等待、命令调度器、日志区
3. 完成初始化同步和状态栏
4. 完成基础时钟控制
5. 完成协议解析与 `device_state`
6. 完成虚拟按键区和协议演示区
7. 完成数字孪生镜像面板
8. 完成 NTP / 天气 / 自动昼夜模式
9. 完成轻量数据看板

这个顺序可以保证先把核心演示链路打通，再补加分项和固件对齐项。
