# 原创功能 Spec：反应时间测试游戏

## 1. 功能定位

在现有时钟 / 闹钟 / 天气 / 对时 / 数字孪生基础上，增加一个可演示、可交互、工作量适中的原创功能：**反应时间测试游戏**。

该功能的目标不是把系统做成完整游戏平台，而是利用现有硬件资源（LED、按键、数码管、串口、PC 上位机）实现一个有明确开始、进行、结束、成绩展示的互动玩法，突出：

- 毫秒级计时能力
- 板端状态机设计
- PC 与 MCU 双向联动
- 日志 / 事件 / UI 同步展示

---

## 2. 设计原则

### 2.1 目标

- 必须和课程要求中的核心功能、扩展功能明显区分
- 必须适合 5 分钟视频演示
- 必须在现有代码结构上可增量实现
- 以作业风格为主，不过度工程化

### 2.2 非目标

首版不追求以下内容：

- 持久化排行榜
- 云端同步 / 联网对战
- 很复杂的难度系统
- 很复杂的异常恢复
- 高精度随机数或防作弊设计
- 花哨动画和皮肤系统

---

## 3. 用户价值

该功能让系统从“被动显示设备”变成“主动交互设备”。

对老师验收的价值：
- 一眼能看出这是原创功能
- 能展示板端逻辑、串口协议、上位机 UI 三者联动
- 结果具有可观测性（反应时间、轮次、平均值）

对学生作业的价值：
- 开发量可控
- 演示效果强
- 易于写入简介 PDF 的“动机-设计-实现-演示”四段

---

## 4. 功能范围

本功能按“完整目标规格”设计，但实现时允许按阶段落地。

### 4.1 首版完整规格（推荐目标）

- 支持 **PC 端启动 / 停止游戏**
- 单次游戏默认为 **5 轮**
- 每轮流程：随机等待 -> 目标点亮 -> 用户按键 -> 成绩判定
- 目标按键使用 4 个主按键：`FUNC / SHIFT / ADD / SAVE`
- 对应 4 个目标 LED（LED1~LED4）
- 记录并显示：
  - 当前轮次
  - 上一轮成绩
  - 最佳成绩
  - 平均成绩
  - 成功次数
- 游戏结束后发送汇总事件给 PC
- PC 端日志记录完整过程

### 4.2 MVP 子集（第一阶段可先做）

- 支持 `START / STOP`
- 只做 1 轮
- 只用 1 个固定按键 + 1 个固定 LED
- 成功后仅显示并上报本次毫秒值

完整规格优先，MVP 作为落地顺序，不作为最终演示上限。

---

## 5. 玩法说明

### 5.1 标准流程

1. 用户在 PC 端点击“开始游戏”
2. MCU 进入游戏状态，清空本局统计
3. 进入第 1 轮，先随机等待 1~3 秒
4. 随机点亮一个目标 LED，并开始计时
5. 用户按下对应按键：
   - 正确且在时限内：记为成功，记录毫秒值
   - 错误按键：记为 miss
   - 超过时限未按：记为 timeout
6. 结果短暂显示后进入下一轮
7. 共完成 5 轮后，输出汇总结果
8. PC 端显示本局统计结果，板端回到正常显示状态

### 5.2 成功与失败判定

- **成功**：用户在目标亮起后，按下与当前目标匹配的按键
- **错误按键**：按下非目标键
- **超时**：目标亮起后 2 秒内未按任何正确键
- **抢按**：在目标亮起前按键
  - 首版可以直接记为 miss
  - 不要求额外复杂处罚机制

## 6. 游戏状态机

建议定义以下状态：

- `IDLE`
  - 非游戏状态，系统保持原正常功能
- `WAIT`
  - 已开始游戏，本轮等待随机触发
- `GO`
  - 目标已经点亮，正在计时，等待用户响应
- `RESULT`
  - 本轮结束，短暂显示本轮结果
- `DONE`
  - 全部轮次结束，生成汇总后退出到 `IDLE`

建议状态流转：

```text
IDLE
  -> START
WAIT
  -> 时间到，选择目标
GO
  -> HIT / MISS / TIMEOUT
RESULT
  -> 如果未满 5 轮，进入下一轮 WAIT
  -> 如果已满 5 轮，进入 DONE
DONE
  -> 输出汇总，回到 IDLE
```

---

## 7. 板端行为规格

## 7.1 LED 行为

游戏期间，LED 显示采用“游戏覆盖”模式，暂时覆盖普通状态 LED 语义。

- `WAIT`：目标 LED 全灭
- `GO`：仅目标 LED 点亮
- `RESULT`：全部熄灭或短闪一次作为结束提示
- `DONE`：可短闪一次后恢复正常 LED 逻辑

说明：
- 游戏过程中允许临时覆盖 DAY / NIGHT 下原本 LED 含义
- 游戏结束后必须恢复现有 LED 逻辑

## 7.2 数码管行为

- `IDLE`：保持原有时间 / 日期 / 消息逻辑
- `WAIT`：可保持当前页，或显示简单等待提示
- `GO`：保持游戏进行态，不强制显示复杂文字
- `RESULT`：短暂显示本轮毫秒值
- `DONE`：短暂显示平均值 / 最佳值中的一个，然后恢复正常显示

为降低实现难度，首版不要求显示复杂英文单词，只要求**结果数值可见**。

## 7.3 按键映射

推荐使用下列映射：

| LED | 目标按键 |
|---|---|
| LED1 | FUNC |
| LED2 | SHIFT |
| LED3 | ADD |
| LED4 | SAVE |

原因：
- 这些键在现有板上已有明确输入路径
- 避免占用 `USER1 / USER2` 的既有扩展联动语义
- 避免影响 `FORMAT / DISP / SPEED` 的原功能含义

说明：
- 游戏态下，这 4 个键临时作为游戏输入
- 游戏结束后恢复其原功能

---

## 8. PC 端行为规格

## 8.1 控制入口

PC 端新增“原创功能 / 游戏模式”分组，至少包含：

- `开始游戏`
- `停止游戏`
- 当前状态显示
- 本次成绩显示
- 最佳成绩显示
- 平均成绩显示
- 当前轮次 / 成功次数显示

## 8.2 日志表现

日志区需要记录以下信息：

- 发送 `*SET:GAME START`
- 游戏事件：开始、目标出现、命中、错误、超时、结束
- 最终汇总结果

日志类型仍沿用现有 send / reply / event / error 分类。

## 8.3 数字孪生联动

首版不强制在 twin panel 里专门增加游戏可视化面板。

最小要求：
- twin panel 按现有事件机制自然跟随 `DISP / LED`
- 游戏成绩与统计显示在控制区或主窗口即可

这样可以减少 UI 改动量。

---

## 9. 协议扩展规格

为兼容现有解析方式，建议所有游戏事件统一使用 `event_name = GAME`。

## 9.1 PC -> MCU 命令

### `*SET:GAME START`
启动一局新游戏。

行为：
- 若当前处于 `IDLE`，开始新一局
- 清空本局统计
- 进入 `WAIT`

### `*SET:GAME STOP`
强制停止当前游戏。

行为：
- 若当前处于游戏中，则立即终止
- 清空临时显示覆盖
- 恢复正常显示 / LED 逻辑
- 返回 `IDLE`

### `*GET:GAME`
查询当前游戏状态。

返回建议：
- `OK IDLE`
- `OK WAIT 2/5`
- `OK GO 3/5 FUNC`
- `OK RESULT 3/5 284`

说明：
- 该查询不是 MVP 必需，但纳入完整规格，便于 PC 首次同步和调试

## 9.2 MCU -> PC 事件

### `*EVT:GAME START`
表示本局开始。

### `*EVT:GAME READY <round> <target>`
表示本轮目标已出现，开始计时。

示例：
- `*EVT:GAME READY 2 FUNC`

### `*EVT:GAME HIT <round> <target> <ms>`
表示本轮成功命中。

示例：
- `*EVT:GAME HIT 2 FUNC 284`

### `*EVT:GAME MISS <round> <target> <actual>`
表示按错键。

示例：
- `*EVT:GAME MISS 2 FUNC SHIFT`

### `*EVT:GAME TIMEOUT <round> <target>`
表示本轮超时。

示例：
- `*EVT:GAME TIMEOUT 2 FUNC`

### `*EVT:GAME DONE <rounds> <success> <best_ms> <avg_ms>`
表示整局结束。

示例：
- `*EVT:GAME DONE 5 4 231 308`

### `*EVT:GAME STOP`
表示游戏被主动停止。

## 10. 数据模型

PC 端建议维护以下游戏状态字段：

- `game_state`：`IDLE / WAIT / GO / RESULT / DONE`
- `game_round_index`
- `game_total_rounds`
- `game_target_key`
- `game_last_result_ms`
- `game_best_result_ms`
- `game_avg_result_ms`
- `game_success_count`
- `game_last_outcome`：`HIT / MISS / TIMEOUT / STOP`

MCU 端建议维护以下字段：

- 当前状态
- 当前轮次
- 总轮次
- 当前目标键 / LED
- 当前轮开始时刻
- 本局累计成功次数
- 本局最佳成绩
- 本局累计成绩求和
- 本轮等待时长

所有统计均只保存在 RAM 中，重启后清空。

---

## 11. 随机规则

首版允许使用简单伪随机：

- 目标键随机从 `FUNC / SHIFT / ADD / SAVE` 中选择
- 等待时间随机在 `1000~3000ms`
- 可基于系统 tick、uptime、轮次进行简单取模生成

不要求真随机。

---

## 12. 与现有功能的兼容策略

### 12.1 与正常时钟显示兼容

- 游戏开始前：保持原功能
- 游戏进行中：显示与 LED 允许临时被游戏覆盖
- 游戏结束后：恢复原功能

### 12.2 与闹钟 / 天气 / 昼夜模式兼容

首版简化策略：

- 游戏期间，游戏显示优先级高于普通页面显示
- 游戏期间，LED 以游戏逻辑为准
- 游戏结束后，回到原有模式

不要求在游戏进行中同时兼顾所有普通页面逻辑。

### 12.3 与虚拟按键兼容

- 游戏开始与停止优先通过 `*SET:GAME` 完成
- 不要求首版通过 `*SET:KEY` 模拟整个游戏流程
- 若后续需要，可允许 PC 通过调试命令模拟目标按键

## 14. 验收标准

以下全部满足即可认为功能完成：

1. PC 端可以启动与停止游戏
2. 板端游戏状态能进入 `WAIT / GO / RESULT / DONE`
3. 板端至少能随机给出目标并计时
4. 用户正确按键后能得到毫秒成绩
5. PC 端能收到并展示游戏事件
6. PC 端能展示本局统计结果
7. 游戏结束后板端恢复正常显示与 LED 逻辑
8. 全流程适合在 60~90 秒内完成演示

---

## 15. 演示脚本建议

建议演示顺序：

1. 打开 PC Host，连接串口
2. 点击“开始游戏”
3. 板端进入游戏态
4. 连续完成 3~5 轮测试
5. 展示 HIT / MISS / TIMEOUT 中至少两种结果
6. PC 端显示最佳值、平均值、成功次数
7. 游戏结束，板端恢复正常时钟显示

---

## 16. 实现建议总结

- 协议格式尽量复用现有 `*SET:* / *GET:* / *EVT:*`
- 事件统一走 `GAME` 主事件名，避免修改整体协议解析框架
- 首版重点是“状态机 + 成绩上报 + UI 展示”三件事
- 不要把工作量花在复杂容错和持久化上

这份文档定义的是**完整目标规格**；实际编码时可先按 MVP 落地，再逐步补齐到本规格。
