# 原创功能实现计划：反应时间测试游戏

## 1. 目标

基于 `docs/原创功能-反应时间测试游戏-spec.md`，为当前项目补充一个可演示、可联调、工作量可控的原创功能：**反应时间测试游戏**。

本计划的目标是：
- 保持当前项目结构不大改
- 分阶段实现 MCU 端状态机与 PC 端展示
- 优先完成“能演示的完整闭环”
- 避免过度工程化，不引入复杂配置、持久化和异常恢复体系

---

## 2. 实现原则

### 2.1 范围控制

本次实现只覆盖课程演示需要的核心功能：
- PC 启动 / 停止游戏
- MCU 进行 5 轮反应测试
- 游戏事件通过串口上报
- PC 端展示当前状态、本次成绩、最佳成绩、平均成绩、成功次数
- 游戏结束后恢复原有时钟系统显示与 LED 逻辑

### 2.2 明确不做

本阶段不做以下内容：
- EEPROM 持久化成绩
- 排行榜
- 网络同步
- 复杂难度系统
- 多页面游戏 UI
- 复杂英文提示和花哨动画
- 特殊失败恢复流程

---

## 3. 实现总阶段

本功能按 5 个阶段推进：

1. **阶段 A：协议与状态骨架**
2. **阶段 B：MCU 游戏运行闭环**
3. **阶段 C：PC 状态解析与展示**
4. **阶段 D：联调与交互修正**
5. **阶段 E：演示收尾与文档核对**

每个阶段都要求有可验证结果，不做一次性大改。

---

## 4. 阶段 A：协议与状态骨架

## 4.1 目标

先把协议和状态字段接入现有代码结构，让 MCU 与 PC 都具备“理解 GAME 功能”的基础能力。

## 4.2 需要修改的文件

### MCU
- `microchip/exp3-1.c`

### PC
- `pc_host/device_state.py`
- `pc_host/commands.py`

## 4.3 MCU 端任务

### 任务 A1：新增游戏状态常量与全局变量
在 `microchip/exp3-1.c` 的全局状态区新增：
- `GAME_STATE_IDLE / WAIT / GO / RESULT / DONE`
- 当前是否处于游戏态
- 当前轮次、总轮数
- 当前目标索引
- 当前轮等待时间
- 当前轮计时值
- 本轮结果
- 最佳成绩
- 累计成绩和成功次数

### 任务 A2：新增函数声明
在函数声明区新增：
- `HandleSetGame`
- `HandleGetGame`
- `ProcessGameTask`
- `ResetGameState`
- `StartGameSession`
- `StartGameRound`
- `HandleGameKeyEvent`
- `FinishGameHit / FinishGameMiss / FinishGameTimeout / FinishGameSession`
- `EmitGameEvent*`
- `GameTargetKeyName / GameTargetLedMask / BuildGameDisplayText`

### 任务 A3：接入 `*SET:GAME` 与 `*GET:GAME`
在 `HandleSetCommand` / `HandleGetCommand` 分发中新增 `GAME` 支持：
- `*SET:GAME START`
- `*SET:GAME STOP`
- `*GET:GAME`

### 任务 A4：在 reset 流程中纳入游戏状态
在协议或系统 reset 时清空游戏相关状态。

## 4.4 PC 端任务

### 任务 A5：在 `DeviceState` 增加游戏字段
增加：
- `game_state`
- `game_round_index`
- `game_total_rounds`
- `game_target_key`
- `game_last_result_ms`
- `game_best_result_ms`
- `game_avg_result_ms`
- `game_success_count`
- `game_last_outcome`

### 任务 A6：在 `commands.py` 增加 `GET:GAME` 识别

当前阶段不强制把 `GET:GAME` 加入初始同步。

## 4.5 阶段验收

完成本阶段后，至少满足：
- MCU 能识别 `*SET:GAME START` / `STOP` / `*GET:GAME`
- PC `DeviceState` 已有游戏状态字段
- 编译 / 运行不因新增字段和声明报错

---

## 5. 阶段 B：MCU 游戏运行闭环

## 5.1 目标

在 MCU 上完成完整的游戏状态机，实现从开始到结束的 5 轮交互闭环。

## 5.2 需要修改的文件

- `microchip/exp3-1.c`

## 5.3 核心任务

### 任务 B1：实现 `StartGameSession`
行为：
- 清空本局统计
- 设置总轮数为 5
- 初始化轮次为第 1 轮
- 发 `*EVT:GAME START`
- 进入 `StartGameRound`

### 任务 B2：实现 `StartGameRound`
行为：
- 切到 `WAIT`
- 生成 `1000~3000ms` 的随机等待时间
- 清理当前轮临时状态
- 刷新 LED / 显示

### 任务 B3：实现 `ProcessGameTask`
按状态推进：
- `WAIT`：等待时间归零后选目标，切到 `GO`
- `GO`：等待用户输入或超时
- `RESULT`：短暂显示后决定下一轮或结束
- `DONE`：结束整局并恢复 `IDLE`

### 任务 B4：在 `SysTick_Handler` 中增加游戏计时
维护：
- `WAIT` 倒计时
- `GO` 计时
- `RESULT` 显示保持时间

### 任务 B5：接管 4 个游戏按键
在 `ProcessKeyTask` 中，当处于游戏状态时：
- `FUNC / SHIFT / ADD / SAVE` 不再走原逻辑
- 改为交给 `HandleGameKeyEvent`

### 任务 B6：实现命中 / 失误 / 超时处理
分别实现：
- `FinishGameHit`
- `FinishGameMiss`
- `FinishGameTimeout`

每个函数都要完成：
- 记录当前轮结果
- 更新统计值
- 进入 `RESULT`
- 发送对应 `*EVT:GAME ...`

### 任务 B7：实现整局结束
在 `FinishGameSession` 中：
- 计算平均成绩
- 发送 `*EVT:GAME DONE <rounds> <success> <best_ms> <avg_ms>`
- 恢复系统普通显示逻辑

## 5.4 游戏事件格式

统一格式如下：
- `*EVT:GAME START`
- `*EVT:GAME READY <round> <target>`
- `*EVT:GAME HIT <round> <target> <ms>`
- `*EVT:GAME MISS <round> <target> <actual>`
- `*EVT:GAME TIMEOUT <round> <target>`
- `*EVT:GAME DONE <rounds> <success> <best_ms> <avg_ms>`
- `*EVT:GAME STOP`

## 5.5 阶段验收

完成本阶段后，至少满足：
- 发送 `*SET:GAME START` 后 MCU 能完整跑完 5 轮
- 正确按键得到 `HIT`
- 错误按键得到 `MISS`
- 不按键得到 `TIMEOUT`
- 最终发送 `DONE`
- 发送 `*SET:GAME STOP` 后可退出游戏

## 6.1 目标

让 PC Host 能正确解析 `GAME` 事件，并在控制面板中显示当前游戏状态与统计信息。

## 6.2 需要修改的文件

- `pc_host/device_state.py`
- `pc_host/widgets/control_panel.py`
- `pc_host/main_window.py`

## 6.3 核心任务

### 任务 C1：在 `DeviceState` 中实现 `reset_game_state`
统一清理所有游戏字段。

### 任务 C2：在 `DeviceState` 中实现 `apply_game_event`
解析 `payload.split()` 后支持：
- `START`
- `READY`
- `HIT`
- `MISS`
- `TIMEOUT`
- `DONE`
- `STOP`

### 任务 C3：把 `GAME` 事件挂入 `apply_event`
当 `event_name == "GAME"` 时，转发给 `apply_game_event`。

### 任务 C4：在 `ControlPanel` 增加游戏区
新增一个游戏分组，包含：
- `开始游戏` 按钮
- `停止游戏` 按钮
- 当前状态标签
- 当前轮次标签
- 当前目标标签
- 本次成绩标签
- 最佳成绩标签
- 平均成绩标签
- 成功次数标签

### 任务 C5：在 `ControlPanel` 中新增 `update_game_state`
从 `DeviceState` 刷新这些标签。

### 任务 C6：在 `MainWindow.refresh_ui()` 调用游戏刷新
在原有：
- `status_bar.update_state`
- `twin_panel.update_state`

之外，再调用：
- `control_panel.update_game_state(self.state)`

## 6.4 阶段验收

完成本阶段后，至少满足：
- PC 能识别 `*EVT:GAME ...`
- 点击开始按钮能发 `*SET:GAME START`
- 日志能看到 GAME 事件
- 控制面板能显示轮次、目标、本次/最佳/平均/成功

## 7. 阶段 D：联调与交互修正

## 7.2 需要修改的文件

### PC
- `pc_host/main_window.py`
- `pc_host/device_state.py`
- `pc_host/widgets/control_panel.py`

## 7.3 核心任务

### 任务 D1：LED 游戏覆盖
在 `ComputeLedOutput()` 中加入游戏优先分支：
- `WAIT`：目标灯灭
- `GO`：仅点亮目标 LED
- `RESULT / DONE`：恢复或简单熄灭

### 任务 D2：数码管显示覆盖
在 `UpdateDisplayBuffer()` 中加入游戏优先分支：
- `WAIT`：空白或简单等待态
- `GO`：可显示轮次或目标信息
- `RESULT`：显示本轮成绩或 MISS/TIMEOUT
- `DONE`：短暂显示平均值或最佳值

### 任务 D3：确保游戏结束后恢复正常功能
验证：
- LED 回到普通意义
- 时钟 / 日期 / 消息页恢复正常
- 按键恢复原功能

### 任务 D4：修正 PC 端 shadow 状态
必要时在 `apply_shadow_from_command()` 中补：
- `*SET:GAME START`
- `*SET:GAME STOP`

仅作为 UI 提前反馈，不作为最终状态来源。

## 7.4 阶段验收

完成本阶段后，至少满足：
- 游戏态时 LED 与数码管表现正确
- 停止或完成后能回到正常时钟系统
- 现有天气 / 编辑 / 普通显示逻辑不因游戏永久损坏

## 10. 推荐开发顺序

严格按以下顺序推进：

1. `microchip/exp3-1.c`：增加状态字段与命令入口
2. `microchip/exp3-1.c`：实现游戏状态机和事件发送
3. `microchip/exp3-1.c`：完成按键接管、LED、数码管游戏覆盖
4. `pc_host/device_state.py`：完成 `GAME` 事件解析
5. `pc_host/widgets/control_panel.py`：完成游戏区按钮和状态标签
6. `pc_host/main_window.py`：接入刷新
7. 联调 MCU 与 PC
8. 最后补测试和演示收尾

理由：
- 先让 MCU 能完整出事件，PC 才有东西可接
- 先做状态闭环，再做 UI 细节
- 这样最容易定位问题

---

## 11. 最终完成标准

当以下条件全部成立时，可以认为原创功能完成：

1. PC 端可启动和停止游戏
2. MCU 能完成 5 轮反应测试
3. 正确按键可获得毫秒成绩
4. 错误按键和超时能区分处理
5. PC 端能实时显示轮次、目标、本次、最佳、平均、成功次数
6. 串口日志中能看到完整 GAME 事件流
7. 游戏结束后系统恢复普通时钟 / LED / 按键逻辑
8. 全流程可在课程演示中稳定展示

---

## 12. 执行建议

本计划不是一次性全部做完的清单，而是阶段推进清单。

建议实际开发时每完成一阶段就停下来做一次验证：
- 阶段 A 看编译和协议入口
- 阶段 B 看 MCU 单侧运行
- 阶段 C 看 PC 是否能正确显示
- 阶段 D 看联调是否有冲突
- 阶段 E 看演示是否顺畅

这样能最大程度避免在作业项目里把问题积累到最后一次性爆发。
